# RIA LinkedIn Prospector — Claude Project Summary (Updated) ## What This Project Is A self-contained web tool for Mike Anderson (AdviceOnly financial advisor) to find and outreach to small RIA (Registered Investment Advisor) firms for referral partnerships. The tool pulls firm data from the SEC public registry, filters by size/location/AUM, surfaces LinkedIn and SEC links for finding individual advisors, and includes **AI-powered email scraping** to find contact emails directly from firm websites. **Live URL:** https://ria-prospector-2.vercel.app **GitHub repo:** mikeanderson-cloud/ria-prospector-2 **Deployment:** Vercel (auto-deploys on GitHub push) --- ## System Architecture ``` ria-prospector/ ├── public/ │ └── index.html ← Entire frontend (single file, ~210KB) ├── api/ │ ├── sec.js ← Vercel serverless proxy for SEC API calls │ └── scrape.js ← NEW: Email scraper — fetches firm websites & extracts emails ├── package.json ← { "type": "module" } └── vercel.json ← Routes /api/sec & /api/scrape to serverless functions ``` **Backend:** Firebase Firestore (project: `the-fox-ria-prospector`) **Auth:** Username + 4-digit PIN, validated against Firestore `users` collection **Progress storage:** Firestore `progress` collection, per user **Email storage:** Firestore `emails` collection, per CRD (firm), with timestamps **Fallback:** localStorage (`ria_v3`) used as local cache; Firestore is source of truth --- ## Authentication & Multi-User System ### Login Flow 1. Every `saveStatus(crd, status)` and `saveNote(crd, note)` call writes to localStorage AND pushes full progress object to Firestore - On login, a Firestore `onSnapshot` listener is attached — any cloud changes sync down to localStorage in real time - Merge logic: cloud wins for any record with newer or equal `updatedAt` - Sync status indicator (colored dot) in header: 🟢 synced / 🟡 syncing / 🔴 error - Export/Import still works as before (JSON + CSV), and also pushes to cloud on import --- ## Email Scraping Feature ### Overview Each firm card includes a **"Find Email"** button that triggers server-side email extraction from the firm's website. ### Data Flow 1. **Website URL source:** Extracted from SEC FOIA CSV during `parseCSV()` parsing - CSV column: "Website Address" - Stored in firm object as `website` field - Baked into `EMBEDDED_FIRMS` list 2. **User clicks "Find Email" button** - Frontend calls `/api/scrape.js?url={firm.website}&crd={firm.crd}` - Serverless function fetches firm's website (not SEC — avoids Cloudflare blocking) - Regex searches for email patterns: `[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}` 3. **Scraper tries multiple endpoints per firm** - `/contact` page - `/about` page - Homepage (root) - Returns first match found or empty array 4. **Results stored in Firestore** - `emails/{crd}` → `{ emails: [...], foundAt: timestamp, source: "scrape", updatedAt }` - Shared across all users (no duplication) - Cached in localStorage for offline access 5. **Card displays emails** - One-click copy-to-clipboard button - Timestamp showing when email was scraped - "Rescrape" button to refresh (stored results won't re-scrape until 7 days pass) ### Why This Approach Works - **No SEC API blocking:** Scraper never touches SEC (which blocks Vercel IPs) - **Targets firm websites directly:** Regular HTML pages, no bot protection - **CSV-driven:** Website URLs come from SEC data upload, not external search - **Shared data:** All users see same emails for a firm (efficiency + consistency) - **Fallback to LinkedIn:** If scrape fails, user can search LinkedIn profiles instead --- ## Key Features ### Landing Screen (after login) -
